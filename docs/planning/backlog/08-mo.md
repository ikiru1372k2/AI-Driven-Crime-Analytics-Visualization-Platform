## Summary
Define the versioned structured MO schema extracted from BriefFacts: attributes, closed vocabularies, UNKNOWN semantics, confidence fields, source-span references — as a JSON Schema used to validate every extraction before persistence.

## Problem Statement
Without a fixed schema, LLM output shape drifts and downstream similarity/anomaly features become uncomparable; arbitrary attributes would be invented data.

## Engineering Objective
`kavach/analytics/mo/schema.py` + `docs/analytics/mo-schema-v1.md`.

## Technical Design (schema v1)
Attributes (each `{value, confidence: 0–1, source_span?: [start,end]}`):
- offender_count: int | "UNKNOWN"
- mobility: enum {on_foot, motorcycle, car, autorickshaw, bicycle, public_transport, other, UNKNOWN}
- approach_method: enum {mobile_approach, stationary_ambush, entry_breakin, deception, confrontation, other, UNKNOWN}
- crime_action: enum {snatching, theft, burglary, robbery, assault, threat, fraud, other, UNKNOWN}
- target_type: enum {gold_chain, mobile_phone, cash, vehicle, jewelry, property, person, other, UNKNOWN}
- escape_direction: free-text | UNKNOWN (display-only, not a similarity feature)
- time_context: enum {night, day, dawn_dusk, UNKNOWN}
- weapon_involved: enum {yes, no, UNKNOWN}
Top-level: model_version, extractor (QUICKML_LLM | RULE_BASED), extracted_at, case_master_id.

## ER Conformance Requirements
Input strictly CaseMaster.BriefFacts; output stored as AI_DERIVED only (never merged into source tables); UNKNOWN mandatory when text lacks evidence.

## Dependencies
Blocked by: {{ER-001}} · Blocks: {{MO-002}}, {{MO-003}}, {{MO-004}}, {{MO-005}}

## Acceptance Criteria
- [ ] JSON Schema published + versioned; validator rejects unknown attributes and out-of-vocabulary values
- [ ] UNKNOWN accepted for every attribute
- [ ] Schema doc includes worked example (chain-snatching narrative → structured output)
- [ ] escape_direction explicitly excluded from similarity feature set (documented)

## Test Plan
Unit: validator accept/reject fixtures.

## Definition of Done
Standard DoD; ADR-006 cross-referenced.

## Demo Evidence
Schema doc shown when judges ask "how do you control the LLM?"

## Limitations / Non-Goals
Extraction itself (MO-002); vocabulary growth is additive versioned change only.

## References
ADR-006; matrix §1.1 (BriefFacts).

## Summary
Integrate Catalyst QuickML LLM serving to extract MO schema v1 from BriefFacts: prompt template demanding schema-conformant JSON with UNKNOWN preference, response validation (MO-001), retry-on-invalid, batch processing, and a deterministic rule-based fallback extractor if QuickML is unavailable/limited (per {{CAT-001}} capability report).

## Problem Statement
BriefFacts narratives are unusable for pattern comparison. LLM extraction without validation risks hallucinated attributes becoming analytical truth — prohibited (ADR-006).

## Why This Matters
Flagship AI capability (C2-R7); MO features feed similarity, anomaly, risk and the D3 demo moment.

## Source Data / ER Schema Mapping
Input: CaseMaster.CaseMasterID + BriefFacts (OBSERVED). Output: schema-v1 JSON (AI_DERIVED) → {{MO-003}} persistence.

## Catalyst Services
QuickML (LLM serving / structured prompt); AppSail (orchestration); Circuits event path later ({{EVT-002}}).

## Dependencies
Blocked by: {{MO-001}}, {{CAT-001}}, {{DATA-002}} · Blocks: {{MO-003}}, {{MO-005}}, {{EVT-002}}

## Technical Design
- Prompt: system template embedding schema + vocabularies + "prefer UNKNOWN over guessing" + few-shot examples (versioned prompt v1, stored in repo)
- Validation loop: parse → JSON Schema validate → on failure retry once with error feedback → on second failure mark case EXTRACTION_FAILED (never partial-persist)
- Confidence: model-reported per attribute if supported, else calibrated defaults documented (do NOT invent per-attribute confidences from nothing — if unavailable, a single extraction_confidence with documented semantics)
- Fallback extractor: keyword/rule-based, same schema, extractor=RULE_BASED, lower default confidence — used only if QuickML unavailable; choice recorded per profile
- Batch: chunked processing with rate-limit backoff; idempotent by (case_master_id, model_version)

## Edge Cases
Empty/1-word BriefFacts (skip, count); very long narratives (truncate with documented window); mixed Kannada/English (v1 scope = English synthetic data — documented; Kannada = FUTURE_INTEGRATION); prompt-injection-style narrative content (output validation contains it); model timeout.

## Acceptance Criteria
- [ ] 100% of persisted profiles validate against schema v1 (invalid → EXTRACTION_FAILED, never stored)
- [ ] Narrative lacking mobility info yields mobility=UNKNOWN (fixture tests)
- [ ] Ground-truth chain-snatching narratives extract expected action/target/mobility on ≥80% of pattern cases (synthetic validation)
- [ ] Fallback path proven by test (QuickML client stubbed unavailable)
- [ ] Every profile carries model_version + extractor + extracted_at

## Test Plan
Unit: validator loop with mocked LLM (valid/invalid/timeout). Integration: real QuickML on 20-case sample (transcript committed). Validation: ground-truth extraction rates ({{MO-005}}).

## Observability
Extraction run metrics: processed / failed / UNKNOWN-rates per attribute (drift indicator).

## Definition of Done
Standard DoD + capability-report citation for QuickML status + no unvalidated output persisted.

## Demo Evidence
D3: click case → narrative alongside extracted structured MO with confidences.

## Limitations / Non-Goals
Not an accuracy claim on real FIRs (no real data); no continuous retraining claims.

## References
ADR-006; {{CAT-001}} report; {{MO-001}}.

## Summary
Compute explainable similarity between MO profiles: weighted attribute agreement over schema v1 (excluding escape_direction), UNKNOWN-aware, producing ranked similar-case lists with per-attribute match breakdown, persisted as MO_SIMILARITY runs.

## Problem Statement
Individual MO profiles don't reveal recurring patterns; analysts need "these 7 FIRs share motorcycle + snatching + gold_chain + night" with the agreement itemized.

## Algorithm / Analytical Method
Similarity(a,b) = Σ w_i · match_i / Σ w_i over attributes where both ≠ UNKNOWN; match = equality (offender_count: banded ±1); weights versioned (`mo-sim v1`, e.g. crime_action 0.25, target 0.2, mobility 0.2, approach 0.15, time_context 0.1, weapon 0.1); minimum 3 comparable attributes else NOT_COMPARABLE; threshold for "similar" 0.75 (versioned). Explainability: per-attribute contribution list. Limitations: no embedding similarity in v1 (documented candidate); pairwise within blocking window (same district cluster or statewide same crime_action) to avoid O(N²).

## Source Data / ER Schema Mapping
MoProfile (AI_DERIVED) → SimilarityEdge (DERIVED, classification POTENTIAL_ASSOCIATION).

## Dependencies
Blocked by: {{MO-003}} · Blocks: {{UI-006}}, {{ENT-002}} (mo_similarity signal), {{GRAPH-001}} (SIMILAR_MO edges)

## API Contract
`GET /api/v1/cases/{id}/mo/similar?limit=` → ranked matches with score + attribute breakdown + classification envelope.

## Edge Cases
Mostly-UNKNOWN profiles (NOT_COMPARABLE, surfaced); ties; self-match exclusion; profiles from different model_versions (compare only same major version — documented).

## Acceptance Criteria
- [ ] Ground-truth recurring MO cases rank each other top-k with score ≥ threshold
- [ ] Unrelated control cases score < threshold
- [ ] Every similarity result includes per-attribute breakdown
- [ ] NOT_COMPARABLE path tested

## Test Plan
Unit: scoring math, UNKNOWN handling. Validation: ground-truth ranking test.

## Definition of Done
Standard DoD.

## Demo Evidence
D3→D4 bridge: "87% MO similarity across 3 FIRs" with breakdown.

## Limitations / Non-Goals
Not identity evidence by itself (feeds ENT as one signal).

## References
{{MO-001}} weights doc; ADR-006.
